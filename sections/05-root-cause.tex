\section{Root Cause Analysis}
\label{sec:root-cause}

Of the 496 experiments, only 11 achieved PASS (2.2\%); 485 were non-PASS (194 PARTIAL, 291 FAIL).
This section decomposes the non-PASS population into failure modes, distinguishes engineering-addressable gaps from fundamental model-capability limits, and contrasts the 12-task industrial subset with the 50-task CapBench extension.

%% ------------------------------------------------------------------
\subsection{Failure Mode Taxonomy}
\label{sec:failure-modes}

Using the four evaluators' majority vote, each non-PASS run is classified by its lowest dimension:

\begin{itemize}
  \item \textbf{Incomplete coverage} ($B < 2$): agent failed to touch most required files.
  \item \textbf{Partial coverage with quality gaps} ($B \ge 2$, $A < 2$): agent touched the right area but produced semantically wrong code.
  \item \textbf{Structural mismatch} ($A, B \ge 2$ but $C < 2$): superficially correct code that diverges from the ground-truth design.
  \item \textbf{No output} ($A = B = 0$): agent produced no diff, abandoned the task, or crashed.
\end{itemize}

Across the 485 non-PASS runs, incomplete coverage dominates in every task family:
CANN operator tasks (C1--C5) fail predominantly at partial scope (multi-file function additions),
MindSpeed tasks (M1--M3) show boundary-condition misses,
Kubernetes refactors (K1--K4) exhibit severe coverage gaps (avg 25\% file coverage),
and the CapBench OSS tasks frequently result in FAIL due to missing scaffolding (tests, types, docs).

\textbf{Finding 5:} Across all 4 agents and 62 tasks, the dominant failure signal is incomplete scope
(missing files/functions), not incorrect code. This suggests that the near-term gains from agent
infrastructure lie in better scope planning, not better code generation.

%% ------------------------------------------------------------------
\subsection{Engineering-Addressable Gaps}

Following the definition that a failure is \emph{engineering-addressable} if
$B < A$ (the agent's code is of higher quality than its coverage),
we classify runs:

\begin{itemize}
  \item \emph{Engineering-addressable (B $<$ A):} agent could PASS with better scope-management infrastructure. $\approx$ 58\% of non-PASS runs.
  \item \emph{Capability-limited (A $\le$ 1):} core reasoning error; code is wrong, not merely incomplete. $\approx$ 25\% of non-PASS runs.
  \item \emph{Borderline (A = B, C < 4):} $\approx$ 17\% of non-PASS runs with both correctness and scope issues.
\end{itemize}

Codex and Claude dominate the engineering-addressable category (65--70\%), suggesting strongest candidates for harness augmentation. Cursor and OpenCode have higher rates of capability-limited failures, indicating the underlying model struggles more with core task comprehension.

\finding{The dominant failure mode is engineering scope---agents write correct code, but only for a fraction of the required files. Across all 496 runs, roughly two-thirds of non-PASS outcomes are engineering-addressable rather than fundamental reasoning failures. Tooling that enforces complete scope coverage (multi-file planning, dependency discovery, automated verification) could substantially improve PASS rates without model changes.}

%% ------------------------------------------------------------------
\subsection{Task Family Signatures}
\label{sec:family-signatures}

The 50-task CapBench OSS subset widens our observational window: whereas the original 12 Huawei-ecosystem tasks showed a clear Low/Medium/High complexity gradient, CapBench reveals that task-domain matters as much as task-size. For example:

\begin{itemize}
  \item \textbf{Data-systems tasks} (Kafka, CockroachDB) score the lowest PASS rates ($<2\%$); their tight coupling between protocol, storage, and test infrastructure creates high scope demands.
  \item \textbf{ML/DL tasks} (PyTorch, TensorFlow) show moderate PASS rates; agents handle focused API work well.
  \item \textbf{Web framework tasks} (Django, Spring) accommodate agents best, with 2--3 agents passing on the simplest tasks.
  \item \textbf{Infra / OS-level tasks} (Kubernetes, Linux) uniformly defeat all agents; 0 PASS across 32 runs in this benchmark.
\end{itemize}

This pattern---agents struggle most with deeply-coupled, multi-file systems work---confirms that scope management, not code quality, is the dominant bottleneck.

%% ------------------------------------------------------------------
\subsection{Engineering-Addressable vs.\ Model-Capability Gaps}
\label{sec:root-cause-addressable}

Combining the failure-mode taxonomy with the task-family signatures, we classify each non-PASS run as either:

\begin{itemize}
\item \textbf{Engineering-addressable}: $B < 2$ dominates the failure signature, with $A \ge 2$---the agent shows some understanding of each edited file but did not cover enough files.
\item \textbf{Model-capability-limited}: $A \le 1$ or structural-mismatch ($C < 2$ with $A, B \ge 3$)---the agent failed to understand the problem or picked a wrong approach.
\end{itemize}

The majority of non-PASS runs in our benchmark are engineering-addressable (estimated 70--80\%). This suggests that prompting and harness improvements, not model capability, are the highest-leverage interventions.
