\section{Discussion and Future Work}
\label{sec:discussion}
\label{sec:industrial-deployment}

The results suggest that current agents should be integrated cautiously: they can draft bounded changes, but they do not yet own system-level feature delivery. We summarize four implications in the style of empirical benchmarking studies.

\paragraph{Implication 1: Route only bounded tasks to agents.}
SE-PASS outcomes appear only on bounded tasks (C1, C4, M1, T10), while the 49 non-trivially scoped main-corpus tasks and all Kubernetes-family tasks produce no SE-PASS. Industrial users should therefore treat agent invocation as a task-shape decision. Local operator changes, focused framework edits, and well-scoped test or registry updates are plausible candidates; cross-component platform features should remain human-authored until a new evaluation demonstrates otherwise.

This implication is deliberately narrower than a blanket adoption claim. The study does not show that agents cannot help with large features; it shows that a single autonomous attempt should not be treated as a merge-ready owner for such work. For broad features, agents may still be useful for exploration, test sketching, or localized subchanges after a human decomposes the task.

\paragraph{Implication 2: Treat structured specifications as engineering artifacts.}
Long prompts improve mean score for every agent (Table~\ref{tab:prompt}), even though they do not solve non-trivial long-horizon tasks. This makes structured task briefs worthwhile as a low-cost intervention. A useful brief should state motivation, expected behavior, constraints, affected subsystems, non-goals, and expected tests before the agent edits code.

The prompt result is especially relevant for industrial teams because it points to process leverage outside the model vendor. A long-form engineering proposal can be reviewed before any code is generated, reused across agents, and attached to the eventual PR as design rationale. Even when the agent output remains partial, the structured brief reduces ambiguity for both the agent and the human reviewer.

\paragraph{Implication 3: Add scope and test checks before human review.}
Table~\ref{tab:failure-signatures} shows that 25.5\% of main-corpus failures are coverage gaps and 18.9\% are borderline partials. These are the failures most likely to be filtered by file-plan checks, required test evidence, type/syntax checks, and subsystem-specific consistency checks. Such checks will not fix semantic failures, but they can prevent incomplete drafts from consuming reviewer attention.

A practical pre-review gate should therefore ask for evidence, not confidence. Examples include a discovered-file plan, an explanation of touched subsystems, a list of tests added or intentionally omitted, and project-specific syntax or type checks. These gates are inexpensive compared with senior review time and directly target the recoverable failure mass identified in \S\ref{sec:addressable-failures}.

\paragraph{Implication 4: Use calibrated multi-evaluator review.}
Pairwise LLM-judge agreement is too low for single-judge evaluation, and same-family bias remains a credible threat in any LLM-as-judge setup. A practical evaluation workflow should use multiple evaluator families, conservative tie-breaking, and periodic human calibration on a representative subset of tasks.

This does not imply that LLM judges can replace code review. The safer interpretation is that calibrated judges can triage agent outputs before they reach reviewers, flagging obvious failures and producing structured rationales. Human review remains necessary for accepted candidates, especially when the implementation is an alternative design rather than a close match to a known reference.

\paragraph{Generalizability to industrial settings.}
The benchmark is intentionally harsher than many day-to-day programming tasks. That harshness is useful for industrial planning: if agents fail at system-level feature ownership, organizations can still deploy them at smaller units while measuring progress on the harder boundary. The findings are therefore most actionable as routing guidance. Teams should maintain a portfolio of eligible task classes, measure agent outputs against local review criteria, and expand eligibility only when pass rates improve under the same calibrated protocol.

\paragraph{Future work.}
Three next steps follow directly from the threats in \S\ref{sec:threats}. First, repeated runs of the SE-PASS-yielding cells would turn the current point estimates into confidence intervals. Second, a stratified human re-rating of main-corpus PARTIAL outputs would estimate the gap between strict SE-PASS and relaxed reviewer-acceptable PASS (RA-PASS). Third, future benchmarks should evaluate agent improvements targeted at the dominant failure modes: long-context semantic reasoning, cross-module registration, test generation, and invariant preservation.
