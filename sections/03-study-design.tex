\section{Methodology}
\label{sec:study-design}

\input{figures/fig_pipeline}

Figure~\ref{fig:pipeline} summarizes the end-to-end protocol. We build a two-tier benchmark, run each task with four commercial agents and two prompt variants, compute a deterministic HW/GT coverage report for each generated diff, then score each run with a calibrated four-evaluator panel.

\subsection{Scope of Tasks}
\label{sec:tasks}

The benchmark contains 62 feature-implementation tasks and 496 agent runs. We use \textbf{GT} to denote the merged ground-truth implementation and \textbf{HW} to denote the handwritten, human-authored subset of GT files after excluding auto-generated artifacts. Each task corresponds to one merged PR and is evaluated at the PR level rather than at the level of a single function or file.

\textbf{Main corpus (LongBench, T01--T50).}
The main corpus consists of 50 long-horizon feature tasks from 15 widely used open-source projects, including Kafka, CPython, Airflow, Kubernetes, Terraform, and Spring. We selected projects that are actively maintained, have public merged PRs with review or CI history, expose tests or documentation that define expected behavior, and span system, framework, platform, and developer-tooling code. The resulting task set provides a reproducible stress test for system-level feature work while avoiding proprietary-source release constraints. We use the term \emph{long-horizon} operationally. A task is long-horizon when the agent must (i) interpret a high-level feature request into an architectural plan, (ii) coordinate edits across more than one subsystem, and (iii) preserve intent over a multi-step trajectory where intermediate feedback is sparse. Feature-enabling refactors are included only when the merged PR changes architecture or infrastructure needed for new behavior, rather than performing a cosmetic rename or mechanical cleanup. The median main-corpus task touches 57 GT files and 6{,}905 lines of diff, but we retain natural outliers at both ends. T10, for example, is a single-file task that remains in the corpus because it was part of the selection process and later serves as an informative sanity check: agents can pass a subsystem-contained task under the same protocol.

\textbf{Industrial calibration subset (C1--C5, M1--M3, K1--K4).}
The second tier contains 12 tasks from an industrial partner's accelerator, distributed-training, and platform-engineering ecosystem. It serves two purposes: (i) human-rater calibration, because three senior reviewers with domain expertise scored all 96 calibration runs; and (ii) discriminant-validity checking, because the subset spans subsystem-contained operator work through Kubernetes-class platform changes. \textbf{Complexity classification.}
For the calibration subset, Low tasks are localized edits with little cross-component coordination, Medium tasks require multi-file changes inside a narrow subsystem, and High tasks require cross-component changes, generated or API-facing surfaces, and stronger domain knowledge. Complexity is therefore a coordination label, not a pure file-count label. This is important for the Kubernetes tasks: K1 is smaller than some accelerator tasks by file count, but it still requires platform-level reasoning about API behavior, validation, and tests.

\subsection{Benchmark Construction}
\label{sec:benchmark-construction}

The main corpus was curated from mature repositories with active maintenance, public merged PRs with review or CI evidence, available tests or documentation, and diversity across systems, frameworks, platforms, developer tools, and languages. We kept PRs that added user-visible behavior or feature-enabling infrastructure and excluded localized bug repairs, dependency bumps, mechanical formatting changes, and behaviorally ambiguous PRs. The industrial calibration subset covers the partner's target stack across low, medium, and high coordination requirements. This two-tier design balances reproducibility with industrial calibration: open-source tasks carry the large-scale aggregate comparison, while industrial tasks test whether the rubric matches expert judgment on deployment-like codebases.

The short prompt approximates the information available in a compact ticket or issue summary. The long prompt approximates a structured engineering brief written before implementation. It can mention affected subsystems at a conceptual level, but it cannot disclose the reference patch's concrete file list. This distinction matters because the study asks whether richer requirements help agents reason about the task, not whether hints from the GT patch can steer them to the answer.

\paragraph{Concrete prompt example.}
Task K3 in \texttt{base\_repo} illustrates the difference. The short prompt asks for digest visibility for Kubernetes image volumes and mentions the \texttt{ImageVolumeWithDigest} gate, \texttt{VolumeMountStatus}, kubelet runtime lookup, gate-off stripping, validation, and tests. The long EP adds the motivation around mutable tags such as \texttt{latest}, then expands the design into API status shape, generated surfaces, Alpha gate behavior, ratcheting-aware drop logic, validation, CRI lookup, and gate-on/gate-off tests. Neither variant includes the GT diff, exact GT file list, or implementation code.

\subsection{Evaluation Metrics}
\label{sec:eval-framework}

Each run is scored on three 0--5 dimensions: \textbf{A} for functional correctness, \textbf{B} for completeness/coverage, and \textbf{C} for behavioral equivalence to the merged implementation. Scores 0--1 indicate a critically deficient dimension, 2--3 indicate partial evidence, and 4--5 indicate reviewable correctness or coverage. We define:
\begin{itemize}
  \item \textsc{se-pass} (strict-equivalence PASS): $A \geq 4$, $B \geq 4$, and $C \geq 3$.
  \item \textsc{fail}: $A < 2$, $B < 2$, or $C < 2$.
  \item \textsc{partial}: all other cases.
\end{itemize}

Before LLM judging, the evaluation harness computes a deterministic HW/GT coverage report for each run. The report splits GT files into HW and auto-generated files, lists covered and missing HW files by intersecting the generated diff with the HW file list, reports GT-file coverage over the full reference file set, and reports function-context coverage over GT hunks inside HW files. These numbers are produced with deterministic file and hunk matching, not LLM judgment. The report is authoritative evidence for the \textbf{B} completeness/coverage score, while the LLM judges still inspect the relevant diffs to decide whether covered files implement the right behavior and whether the output is behaviorally equivalent to GT. In the analysis, we report HW-file coverage, GT-file coverage, function-context coverage, and LLM A/B/C scores as separate sub-scores rather than collapsing deterministic coverage and LLM judgment into a fixed weighted index.

SE-PASS is intentionally stricter than test passing. It asks whether the agent's PR is close enough to the human-merged PR in functionality, file organization, API surface, and test structure to be considered merge-ready. This strict-equivalence rule can mark valid alternative implementations as non-SE-PASS, so the headline pass rates should be read as conservative lower bounds on merge readiness. RA-PASS, used only in the threats discussion, denotes a relaxed reviewer-acceptable verdict that would allow such alternatives.

We do not use test execution as the headline metric because replaying 496 candidate diffs would require substantial CI reconstruction across dependency caches, external services, accelerator environments, and project-specific sharding. Suite-level pass/fail is also too coarse for feature PRs, so we score expected tests and check-bearing structure as part of the coverage dimension while leaving full CI replay to future task-specific harnesses.

The threshold is chosen to match reviewer risk. Functional correctness and coverage both need scores of at least 4 because a feature PR with a missing core behavior or missing subsystem integration is not review-light, even if the visible code is plausible. Behavioral equivalence uses a lower threshold of 3 because industrial reviewers may accept a different internal design when it preserves compatibility, API shape, and observable behavior. This asymmetry is why SE-PASS is stricter than a simple mean-score cutoff.

Every run is independently scored by four LLM evaluators from different model families: Claude Opus~4.6, Codex GPT-5.4, Cursor Composer-2, and GLM-5.1. Each evaluator receives the deterministic coverage report together with the GT diff, generated diff, and task prompt. We report the majority verdict with a conservative tie-break that orders FAIL$<$PARTIAL$<$SE-PASS and chooses the lower label when evaluators split. This design addresses known single-judge instability and self-preference risks in LLM-as-judge evaluation~\cite{gu2024llmjudgesurvey,thakur2024judgingjudges,jung2024trustescalate,wataoka2024selfpreference}.

\textbf{Human calibration.}
Three blinded senior reviewers scored all 96 calibration-subset runs under the same rubric. Human verdicts were 9~SE-PASS / 55~PARTIAL / 32~FAIL; the LLM panel produced 4~SE-PASS / 57~PARTIAL / 35~FAIL. SE-PASS-vs-non-SE-PASS agreement is 92.7\% ($\kappa{=}0.43$). The panel is stricter: it demoted 6 of 9 human-SE-PASS cells and never promoted a human-FAIL.

We use the LLM panel for scale and the human subset for calibration, not as interchangeable authorities. The calibration result supports the headline conclusion because the disagreement is conservative with respect to SE-PASS: the automated panel is more likely to demote a human-accepted run than to promote a rejected one. For this reason, the reported SE-PASS rates should be read as lower-bound estimates of strict merge readiness rather than optimistic automation claims.

\subsection{Implementation Details}
\label{sec:execution}

Each task--agent--prompt cell runs once in a fresh sandbox at the parent commit of the target PR. Upstream remotes are removed, network access to upstream repositories is blocked, and \texttt{gh}/\texttt{git fetch} paths are shimmed to prevent the agent from recovering the GT commit. This anti-leakage protocol was added after an early run on a Kubernetes task surfaced the reference patch through repository history; all reported numbers use the fixed protocol.

Each run ends with the diff produced by the agent in the sandbox. We preserve the raw diff, command transcript when available, evaluator prompts, per-judge scores, majority verdict, and sanitization record. We do not manually repair agent outputs before judging. This keeps the unit of analysis aligned with the industrial workflow question: what arrives at review after a single autonomous agent attempt?
