\section{Study Design}\label{sec:study-design}

\input{figures/fig_pipeline}

Figure~\ref{fig:pipeline} summarizes our experiment pipeline.
We design a controlled experiment to assess coding agent capabilities on
real feature implementation tasks drawn from production repositories.
This section describes the task selection, agent configurations, prompt
design, evaluation framework, and experiment execution protocol.

%% ------------------------------------------------------------------
\subsection{Task Selection}\label{sec:tasks}

Our benchmark comprises two complementary tiers.

\textbf{Main evaluation corpus (CapBench, T01--T50).}
The primary tier consists of 50 long-horizon feature-implementation tasks drawn from 15 widely used open-source projects (Kafka, CPython, Airflow, Kubernetes, Terraform, Spring, etc.) spanning six languages (Java~20, Python~13, Go~8, C++~6, TypeScript~2, Rust~1). Each task corresponds to a single merged pull request implementing one coherent feature or refactor; the median task touches 57 ground-truth files and 6{,}905 lines of diff. The full CapBench task inventory is included in the replication package.

We use the term \emph{long-horizon} in the sense crystallized across recent coding-agent benchmarks~\cite{swebenchpro2025, sweevo, featbench2025}: a task is long-horizon when it requires the agent to (i)~interpret a high-level requirement into an architectural plan spanning multiple modules, (ii)~coordinate edits across many files while preserving cross-file invariants, and (iii)~sustain coherent intent over a prolonged action trajectory in which intermediate feedback is sparse and errors compound. Our CapBench tasks have a median of 57 ground-truth files changed and 6{,}905 lines of diff---an order of magnitude beyond SWE-Bench Pro (avg 4.1 files, 107~LOC) and two orders of magnitude beyond SWE-bench Verified (avg 1.7 files, 33~LOC).

\textbf{Calibration subset (C1--C5, M1--M3, K1--K4).}
The second tier consists of 12 industrial tasks from the Huawei/platform ecosystem---the CANN/\texttt{torch\_npu} accelerator stack, MindSpeed, and Kubernetes---summarised in Table~\ref{tab:tasks}. These tasks serve two roles: (i)~\emph{human-rater pilot calibration}: 96 runs (12~tasks $\times$ 4~agents $\times$ 2~prompts) were scored by three blinded senior reviewers with domain expertise in these codebases, producing 92.7\% SE-PASS-vs-non-SE-PASS agreement with the LLM panel (\S\ref{sec:eval-framework}); and (ii)~\emph{rubric discriminant validity}: simple bounded tasks (Low/Medium) achieve \textsc{se-pass} while cross-cutting Kubernetes work (High) does not, confirming the rubric separates difficulty levels as intended.

\textbf{Combined benchmark.}
Combining both tiers gives \textbf{62 tasks $\times$ 4 agents $\times$ 2 prompt variants = 496 runs}.
Figure~\ref{fig:study-arch} summarizes how the two tiers interact: the calibration set validates the evaluation pipeline, while the primary experiment measures the capability boundary and derives the deployment gates.

\textbf{Calibration-subset detail.}
Every task in both tiers was drawn from a real, merged contribution to a production
repository. Within the calibration subset, C-family accelerator tasks (C1--C5) and MindSpeed tasks (M1--M3) come from
Huawei's Ascend NPU software stack---operator kernels, \texttt{torch\_npu}
integration, training framework features, and precision fixes that
represent typical industrial workloads. Kubernetes tasks (K1--K4) come from the
Kubernetes project and cover KEP-driven feature gates, API types,
controllers, and authorization changes, representing large-scale
open-source engineering.

\textbf{Complexity classification.}
We classify tasks into three tiers based on the number of ground-truth
files, cross-component coordination requirements, generated-code or API
surface, and domain-specific knowledge needed. The thresholds are not
pure file-count bins: a small Kubernetes API change can be High because
it requires generated types, validation, tests, and release-cycle
knowledge, while a bounded Python change can remain Medium even with a
larger diff.
\begin{itemize}
  \item \emph{Low} (C1, M1): localised edits with limited architectural
    coupling.
  \item \emph{Medium} (C2, C3, M2, M3): bounded multi-file changes or
    domain-specific implementation work that remains inside a narrow
    subsystem.
  \item \emph{High} (C4, C5, K1--K4): cross-component changes that
    require architectural understanding, generated/API surface updates,
    multi-module orchestration, or extensive testing.
\end{itemize}

The calibration-subset tasks span four languages (C++, AscendC, Python, Go) and all three complexity tiers.

\paragraph{CapBench selection criteria.}
Each CapBench task was selected under three criteria: (i)~the merged PR adds new behaviour rather than fixing a regression, (ii)~the change touches at least two source modules, and (iii)~each task ships with the upstream PR's test suite as the behavioural oracle.

\textbf{Product-level comparisons.}
Each configuration pairs a different model with a different agent framework,
so comparisons confound model capability with framework engineering.
These comparisons are best understood as \emph{product-level}
assessments---comparing the capabilities of commercially available agent
tools---rather than controlled model ablations.
We discuss this confound further in \S\ref{sec:threats}.

\subsection{Prompt Design}\label{sec:prompts}

Each task is paired with two prompt variants that differ in the level
of specification detail provided to the agent:

\begin{itemize}
  \item \textbf{Short prompt}: contains a \emph{Summary} (one-sentence
    problem statement) and a \emph{Proposal} (one-sentence solution
    direction).  Typically 2--4 sentences.
  \item \textbf{Long prompt}: extends the short prompt with
    \emph{Motivation} (why the change is needed) and \emph{Design
    Details} (numbered list of implementation considerations).
    Typically 15--30 sentences.
\end{itemize}

\textbf{Design principle.}
Both prompt levels describe \emph{what} to implement and \emph{why},
but never reveal specific file paths, function names, or code
snippets from the ground truth.  This ensures that agents must
independently navigate the codebase to locate relevant files and design
their implementation.

\textbf{Example (C1, buffer alias fix).}
The short prompt reads: ``\emph{ScaledMaskedSoftmaxGradV2 has a
tensor-lifetime bug: a buffer is used after its queue resource is freed.
Replace the reference with an equivalent surviving buffer.}''  The long prompt adds three numbered design
details: root cause analysis of the data flow, the fix strategy
(identify a shape/dtype-compatible live buffer), and the impact scope
(NormHeadDim variant only).

This two-level design allows us to measure the \emph{prompt
granularity effect}: the extent to which richer specification
compensates for agent limitations in codebase understanding and
architectural reasoning.

\subsection{Evaluation Framework}\label{sec:eval-framework}

\textbf{Ground truth.}
For each task, the ground truth is the actual merged code from the
repository (commit or pull request).  This provides a concrete, human-reviewed
reference implementation against which we compare agent output.

\textbf{Scoring dimensions.}
Each experiment is scored by an LLM evaluator on three dimensions, each
on a 0--5 scale:
\begin{itemize}
  \item \textbf{Score~A (Functional Correctness)}: Does the generated
    code correctly implement the intended functionality?
  \item \textbf{Score~B (Completeness \& Coverage)}: Does the
    implementation cover all required files and components?
  \item \textbf{Score~C (Behavioral Equivalence)}: Does the
    implementation match the ground truth's design decisions, API
    conventions, and architectural approach?
\end{itemize}

\textbf{Verdict rules.}
We assign a three-level verdict based on the dimension scores:
\begin{itemize}
  \item \textsc{se-pass} (strict-equivalence PASS): $A \geq 4$ \textbf{and} $B \geq 4$ \textbf{and} $C \geq 3$.
  \item \textsc{fail}: $A < 2$ \textbf{or} $B < 2$ \textbf{or} $C < 2$ (any dimension critically deficient).
  \item \textsc{partial}: all other cases.
\end{itemize}

We deliberately adopt a strict-equivalence criterion rather than a relaxed ``functionally correct'' criterion, because our deployment question is whether an agent draft can \emph{replace} a human-authored PR, not merely whether it contains a correct local fix. SE-PASS therefore requires not only functional correctness but also alignment with the reference implementation in file organisation, API surface, and test structure. This means our SE-PASS rate is a conservative lower bound on merge-readiness: valid alternative implementations that a reviewer might accept are counted as non-SE-PASS under this rubric.

\textbf{File coverage metric.}
We additionally compute file-level overlap between the agent's generated
files and the ground-truth file set:
$\mathit{coverage} = |\mathit{generated} \cap \mathit{GT}| \;/\;
|\mathit{GT}|$.  This provides a coarse quantitative complement to the
expert-scored dimensions.

\textbf{Evaluation scope.}
Our evaluation measures \emph{production-level equivalence} to the merged implementation, not purely functional correctness.
An agent solution that correctly addresses the requirement but uses a materially different file organization, naming convention, or architecture than the accepted PR will score lower on Behavioral Equivalence~(C) and file coverage.
This design choice reflects our industrial interest: we assess whether agent output could replace a human-authored PR draft, not merely whether it contains a plausible local fix.
As a consequence, our \textsc{fail} verdicts should be read as ``not production-equivalent'' rather than ``functionally impossible.''
Our root-cause analysis (\S\ref{sec:root-cause}) confirms this: many non-SE-PASS cases stem from engineering completeness gaps rather than total semantic misunderstanding.

\textbf{Multi-evaluator cross-validation.}
To mitigate evaluator bias, every experiment is independently scored by
four LLM evaluators from different model families: Claude Opus~4.6,
Codex GPT-5.4, Cursor Composer-2, and GLM-5.1.  We report the
\emph{majority verdict} (strict majority where possible, with a conservative
tie-break that orders labels FAIL$<$PARTIAL$<$SE-PASS and selects the lower label
when evaluators are split) and scores averaged across the four.  Unanimity on the
verdict label holds for 33.7\% of experiments; pairwise verdict agreement
ranges from 42.5\% (Codex vs GLM, the two strictest evaluators) to 74.4\%
(Claude vs Cursor).  When the four evaluators split evenly between SE-PASS and
non-SE-PASS (a 2--2 tie), our tie-break records the run as non-SE-PASS, which
keeps headline SE-PASS rates conservative.

\textbf{Human pilot calibration.}
To check that the LLM panel is not idiosyncratically biased relative to
domain-expert review, we conducted a blind human-rater pilot on the 96
calibration-subset runs (12 calibration tasks $\times$ 4 agents $\times$
2 prompts) because the senior reviewers had domain expertise in these
Huawei codebases.  Three raters scored each run under the same rubric
(A/B/C dimensions, identical SE-PASS/PARTIAL/FAIL verdict rules) without
seeing LLM-panel scores.  On these 96 cells the
human verdicts came back as \textbf{9~SE-PASS / 55~PARTIAL / 32~FAIL}, vs
the LLM-panel \textbf{4~SE-PASS / 57~PARTIAL / 35~FAIL} under our 3-of-4
majority rule; raw three-way verdict agreement is 66.7\% (Cohen's
$\kappa = 0.38$), and SE-PASS-vs-non-PASS agreement is 92.7\% ($\kappa = 0.43$).
The LLM panel is the stricter of the two: 6 of the 9 human-SE-PASS cells
were demoted by the panel (4 to PARTIAL, 2 to FAIL), and the panel never
overturned a human-FAIL into an SE-PASS. The qualitative pattern is identical:
both methods see well under 10\% SE-PASS on the calibration subset (humans
9.4\%, panel 4.2\%), far below the threshold that would change the
deployment-readiness conclusion. We treat this as evidence that the
1.0\% headline SE-PASS rate on the main corpus (1.6\% across all 496 runs) is a
conservative \emph{lower} bound on what a careful human reviewer would mark merge-ready, not a strict-evaluator artifact: humans were more lenient than the panel on a handful of borderline runs, never harsher.  Per-rater
rationales are released in the artifact bundle.

\subsection{Experiment Execution}\label{sec:execution}
Each (task, agent, prompt) cell runs once in a fresh sandbox built per task. The repository is rebased to the parent commit of the merged PR, with upstream remotes removed and \texttt{gh}/\texttt{git fetch} shimmed off, so the agent cannot recover ground truth from history. We discovered the importance of the second precaution after an early Codex run on K4 reached the merged patch via \texttt{git log}; we now treat anti-leakage as a benchmark requirement rather than an afterthought.

\subsection{Experiment Execution (legacy long form)}\label{sec:execution-legacy}

\textbf{Sanitization protocol.}
To prevent agents from extracting ground-truth code from repository
history, we apply a strict sanitization procedure to each experiment
repository:
\begin{enumerate}
  \item The repository is rebased to a single commit immediately before
    the target feature, removing all commits that contain ground-truth
    code.
  \item All git remotes are stripped and the \texttt{gh} CLI is blocked,
    preventing agents from fetching upstream history or pull request
    contents.
  \item Agents retain access to the full codebase (for context) and to
    web search (for documentation), but cannot access the specific PR or
    commit that constitutes the ground truth.
\end{enumerate}

\textbf{Anti-cheating audit.}
During evaluation of initial Codex runs on task~K4, we discovered that
the agent had executed \texttt{git log} and \texttt{git cherry-pick}
commands to extract ground-truth commits from an insufficiently
sanitized repository.  The K4 repository was re-sanitized (all target
commits removed from history) and the affected experiments were re-run.
Post-sanitization results were confirmed genuine through inspection of
agent trajectories and comparison of generated file counts and
implementation details against ground truth.
This incident underscores a critical finding: frontier models actively
search for shortcuts in version control history, making thorough
sanitization essential for any benchmark using real repositories.
