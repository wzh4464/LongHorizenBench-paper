\section{Threats to Validity}
\label{sec:threats}

\subsection{Internal Validity}

Each task--agent--prompt cell was run once, so stochastic decoding and tool-use variation could move borderline outputs across verdict thresholds; we therefore report distributions and large effects, not only pass rates, and treat SE-PASS rates as point estimates. Because each evaluated product combines a model, tools, planning loop, and hidden validation behavior, agent differences are product-level observations rather than controlled model-vs-harness causal claims. Anti-leakage remains a risk: after one early run recovered a target patch from repository history, all reported runs used parent commits, removed remotes, shimmed fetch paths, and the fixed protocol. Human calibration also carries subjectivity despite blinded senior reviewers and a shared rubric; their ratings calibrate the SE-PASS boundary rather than exhaustively reproducing all review judgments.

\subsection{Construct Validity}

SE-PASS is a strict-equivalence construct requiring functional correctness, coverage, and behavioral alignment with the merged reference implementation. This matches the question of whether an agent can replace a human-authored PR, but it can reject valid alternative designs a reviewer might accept; we therefore treat SE-PASS as a conservative lower bound and leave RA-PASS as a future relaxed reviewer-acceptable construct. LLM-as-judge evaluation may introduce scoring bias, disagreement, and same-family/self-preference bias~\cite{wataoka2024selfpreference}; we reduce this risk with deterministic HW/GT coverage reports, four independent evaluators, conservative tie-breaking, leave-one-family-out analysis, and human calibration, but a stratified human re-rating of main-corpus runs would further strengthen the transfer assumption. Because the merged PR is not the only possible implementation, the binary SE-PASS label should be read with mean A/B/C scores and failure signatures.

\subsection{External Validity}

The main corpus comes from 15 open-source projects and the calibration subset from one industrial partner's accelerator, distributed-training, and platform-engineering stack. The benchmark intentionally stresses long-horizon, system-level feature work, so results should not be generalized to all programming tasks, smaller bug fixes, test-only changes, CRUD applications, data-pipeline maintenance, or interactive pair-programming workflows with frequent human feedback. Agent versions are fixed to the study period and may improve, so we release the benchmark and scoring artefacts to support reruns. The scoped claim is that current agents are not ready for autonomous industrial feature delivery in system-level settings of this kind.
