Frontier coding agents now solve a large fraction of single-file bug-fix benchmarks, but industrial teams mainly need help with multi-file feature implementation: API changes, cross-module wiring, generated artifacts, and tests that must land as one coherent pull request.
We report an industrial capability assessment designed to answer a deployment question faced by Huawei engineering teams: which feature tasks, if any, can current commercial agents complete autonomously?
We evaluate four coding agents---Claude Opus~4.6, GPT-5.4 (Codex), Composer-2 (Cursor Agent), and GLM-5.1 (OpenCode)---on 62 real feature-implementation tasks: 12 from Huawei CANN, MindSpeed, torch\_npu, and Kubernetes workflows, plus 50 open-source CapBench tasks spanning Kafka, CPython, Airflow, TensorFlow, Django, and other production projects.
Each of the $4 \times 62 \times 2 = 496$ task--agent--prompt combinations is independently scored by four LLM judges using a three-dimension production-equivalence rubric (functional correctness, completeness, behavioral equivalence), then combined by majority vote with conservative tie-breaking.
Across 496 experiments, only 11 runs PASS (2.2\%); the current best configuration, Cursor Composer-2, passes 3.2\%, and OpenCode GLM-5.1 passes 0.8\%.
Three patterns emerge: (i) a complexity cliff, with 0\% PASS on every High-complexity task; (ii) a systems-code cliff, with Kubernetes refactors yielding 0\% PASS across 32 runs while the CANN family reaches 12.5\%; and (iii) limited prompt leverage, where long specifications lift Claude and Codex from 0\% to 4.8\% PASS but do not rescue multi-module tasks.
The results argue against autonomous deployment for feature work today, but identify a practical path forward: restrict agents to small, reviewable tasks and invest in harnesses that enforce scope discovery, test generation, and multi-evaluator acceptance.
