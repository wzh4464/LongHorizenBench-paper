LLM-based coding agents now solve a large fraction of single-file bug-fix benchmarks, prompting industry interest in deploying them for multi-file feature implementation.
Yet no rigorous, multi-repository benchmark exists for this setting.
We evaluate four state-of-the-art coding agents---Claude Opus~4.6, GPT-5.4 (Codex), Composer-2 (Cursor Agent), and GLM-5.1 (OpenCode)---on 62 real feature-implementation tasks drawn from production repositories: 12 industrial tasks from Huawei CANN, MindSpeed, torch\_npu, and the upstream Kubernetes project, plus 50 open-source tasks (CapBench) spanning Kafka, CPython, Airflow, TensorFlow, Django, and 10 other well-known projects in C, C++, Go, Java, and Python.
Each of the $4 \times 62 \times 2 = 496$ task--agent--prompt combinations is independently scored by four LLM judges using a three-dimension rubric (functional correctness, completeness, behavioral equivalence), combined by majority vote with a conservative tie-break.
Across 496 experiments, only 11 PASS (2.2\%); the best agent (Cursor Composer-2) passes 3.2\% of runs and the weakest (OpenCode GLM-5.1) 0.8\%.
Three patterns emerge: (i) an absolute complexity cliff---0\% PASS on any High-complexity task for any agent; (ii) an OS-vs-framework cliff---Kubernetes refactors yield 0\% PASS across 32 experiments while smaller CANN tasks retain 17.5\% PASS; (iii) prompt specificity is the single strongest lever for top agents (Codex/Claude jump from 0\% on short prompts to 4.8\% on long prompts), while it saturates for weaker agents.
The results indicate that current coding agents are not ready for autonomous multi-file feature implementation; their production use requires either prompt-scaffolded human oversight or restricting scope to small, self-contained tasks.
