Industrial teams now face a deployment question, not a benchmark question: \emph{can current coding agents produce feature pull requests}---multi-file, API-changing, test-bearing changes---\emph{that engineers would actually merge?} We answer this for Huawei's rollout by auditing four leading commercial agents on 62 real feature tasks: 12 from Huawei's CANN/\texttt{torch\_npu}, MindSpeed, and Kubernetes work, plus 50 from our open-source extension CapBench. Across $4 \times 62 \times 2 = 496$ runs (long and short prompts) judged by a four-LLM panel calibrated against three blinded human reviewers (92.7\% PASS-vs-non-PASS agreement, panel runs strictly stricter), only $1.6\%$ of runs reach a merge-ready verdict, and the four Kubernetes tasks return $0$ of $32$ PASS for every agent. We harden the audit against ground-truth leakage after observing one agent cherry-picking the target diff from upstream history. The 1.6\% headline is not the contribution: the contribution is the deployment policy our team now runs in production at Huawei---a four-lever routing pipeline that (i) gates incoming tickets to an allow-list of bounded operator/refactor tasks, (ii) requires a long-form specification template before any agent run, (iii) blocks agent drafts at a pre-review CI gate enforcing declared file scope and test coverage, and (iv) admits drafts to human review only after a 3-of-4 LLM-judge acceptance panel. The pipeline keeps cross-cutting changes (Kubernetes-class controllers, protocol-level refactors) under human authorship until the gates close on a measurable failure mode.
