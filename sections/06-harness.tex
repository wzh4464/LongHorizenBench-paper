\section{Harness Engineering}
\label{sec:harness}

The previous sections established that no agent configuration reliably completes industrial feature implementation (Section~\ref{sec:results}) and that a large engineering-addressable share of failures involves incomplete file and test coverage (Section~\ref{sec:root-cause}).
This section examines how much of the gap can be addressed by \emph{harness engineering}: changing the task scaffold, execution loop, and acceptance checks around an agent without changing the underlying model.

%% ──────────────────────────────────────────────────────────────────
\subsection{Definition and Taxonomy}
\label{sec:harness-def}

We define \textbf{Harness Engineering} as the practice of augmenting a coding agent's execution environment with structured inputs, validation loops, or process constraints---without modifying the underlying model.
We distinguish three intervention points:

\begin{itemize}
  \item \textbf{Input-side} (specification harness): detailed task descriptions, file-level scope constraints, or architecture guidelines provided before generation begins.  Our long prompts serve this role.
  \item \textbf{Output-side} (validation harness): syntax checking, structure validation, and test execution that feed errors back to the agent for correction. Our main experiment does not include this level.
  \item \textbf{Process-side} (pipeline harness): enforced phases such as discovery-before-implementation, mandatory test generation, or CI-gated iteration. Our main experiment does not include this level.
\end{itemize}

%% ──────────────────────────────────────────────────────────────────
\subsection{Input-Side Harness: The Prompt Effect}
\label{sec:harness-prompt}

Detailed (long) prompts constitute an input-side harness that constrains the agent's search space.
Table~\ref{tab:prompt} (in \S\ref{sec:results-prompt}) shows the effect across all four configurations.

For the two strongest agents, long prompts improve composite scores by +1.2 to +1.7/15 and pass rates by 4.8 percentage points.
Claude and Codex show similar prompt deltas (+4.8~pp), passing 4.8\% of tasks with long prompts vs.\ 0\% with short prompts.
However, the effect is not universal: Cursor and OpenCode each reach only 1.6\% PASS with long prompts and 0\% with short prompts.

\finding{Detailed specifications improve PASS rates by up to 4.8 percentage points and composite scores by +1.2 to +1.7/15 for the two strongest configurations.
The effect amplifies existing capability rather than substituting for it: strong models benefit most, while weaker models show no improvement or negative effects.}

\subsection{Harness Limitations and Future Directions}
\label{sec:harness-limits}

Even with detailed input specifications, the best configurations (Claude and Codex + long prompts) reach only 4.8\% PASS.
The remaining gap stems from:
\begin{itemize}
  \item \textbf{Test generation:} No harness enforces test writing; agents frequently omit test files that constitute a significant fraction of the ground truth.
  \item \textbf{Architecture reasoning:} Input specifications can guide scope but cannot verify that cross-module wiring is semantically correct.
  \item \textbf{Discovery depth:} The harness does not control how thoroughly agents explore the codebase before writing code.
\end{itemize}

The data supports input-side harnessing directly. The following two
levels are roadmap items motivated by observed failures rather than
measured ablations:
\label{sec:harness-roadmap}
(1)~\emph{language-aware validation}, adapting output checks to Go/Python/C++ conventions;
(2)~\emph{fix-cycle loops}, feeding validation errors back for agent correction; and
(3)~\emph{architecture-aware planning}, enforcing discovery-before-implementation phases and explicit test plans.

\finding{Input-side prompt engineering lifts Claude and Codex from 0\% to 4.8\% PASS and Cursor/OpenCode from 0\% to 1.6\%. Closing the remaining >95\% gap requires output-side and process-side harnesses---fix-cycle loops and enforced test coverage---that no current agent implements natively.}

\paragraph{Deployment recipe for near-term production use.}
For practitioners shipping coding agents to internal developers today, our data suggests four low-cost hardening steps.
(i) \emph{Require a long-form task spec} before agent invocation; the $+$2--6~pp PASS uplift we measure for long prompts is the single largest free-lunch effect in the study.
(ii) \emph{Gate outputs on syntax + unit tests} in a short post-agent loop; coverage gaps are the clearest harness-addressable failure bucket, so this is cheap.
(iii) \emph{Declare scope explicitly}---list the directories the agent may touch---to constrain the edit set and avoid the common failure of stopping before all required files are identified.
(iv) \emph{Run an evaluator panel} (minimum two evaluator models) before merging; we observed 42\%--75\% pairwise agreement between judges, so single-judge gating is noisy.
