\section{Harness Engineering}
\label{sec:harness}

The previous sections established that no agent configuration reliably completes industrial feature implementation (Section~\ref{sec:results}) and that failures cluster around engineering-addressable gaps---primarily incomplete file coverage (Section~\ref{sec:root-cause}).
This section examines whether engineering interventions---specifically, detailed input specifications---can close the performance gap.

%% ──────────────────────────────────────────────────────────────────
\subsection{Definition and Taxonomy}
\label{sec:harness-def}

We define \textbf{Harness Engineering} as the practice of augmenting a coding agent's execution environment with structured inputs, validation loops, or process constraints---without modifying the underlying model.
We distinguish three intervention points:

\begin{itemize}
  \item \textbf{Input-side} (specification harness): detailed task descriptions, file-level scope constraints, or architecture guidelines provided before generation begins.  Our long prompts serve this role.
  \item \textbf{Output-side} (validation harness): syntax checking, structure validation, and test execution that feed errors back to the agent for correction.  No configuration in our study implements this level.
  \item \textbf{Process-side} (pipeline harness): enforced phases such as discovery-before-implementation, mandatory test generation, or CI-gated iteration.  No configuration in our study implements this level.
\end{itemize}

%% ──────────────────────────────────────────────────────────────────
\subsection{Input-Side Harness: The Prompt Effect}
\label{sec:harness-prompt}

Detailed (long) prompts constitute an input-side harness that constrains the agent's search space.
Table~\ref{tab:prompt-effect} (in \S\ref{sec:results-prompt}) shows the effect across all four configurations.

For the two strongest agents, long prompts improve composite scores by +2.2 to +2.6/15 and pass rates by 25.0 percentage points.
Both Claude and Codex show identical prompt deltas (+25.0~pp), passing 25.0\% of tasks with long prompts vs.\ 0\% with short prompts.
However, the effect is not universal: Cursor shows a \emph{negative} delta ($-$16.7~pp), and OpenCode shows no change (0\% regardless of prompt detail).

\finding{Detailed specifications improve PASS rates by up to 25.0 percentage points and composite scores by +2.2 to +2.6/15 for the two strongest configurations.
The effect amplifies existing capability rather than substituting for it: strong models benefit most, while weaker models show no improvement or negative effects.}

\subsection{Harness Limitations and Future Directions}
\label{sec:harness-limits}

Even with detailed input specifications, the best configurations (Claude and Codex + long prompts) reach only 25.0\% PASS.
The remaining gap stems from:
\begin{itemize}
  \item \textbf{Test generation:} No harness enforces test writing; agents frequently omit test files that constitute a significant fraction of the ground truth.
  \item \textbf{Architecture reasoning:} Input specifications can guide scope but cannot verify that cross-module wiring is semantically correct.
  \item \textbf{Discovery depth:} The harness does not control how thoroughly agents explore the codebase before writing code.
\end{itemize}

We identify three levels of harness improvement, ordered by estimated difficulty:
\label{sec:harness-roadmap}
(1)~\emph{language-aware validation} (adapt output checks to Go/Python/C++ conventions; estimated +2--5\% PASS);
(2)~\emph{fix-cycle loops} (feed validation errors back for agent correction; estimated +10--15\% PASS); and
(3)~\emph{architecture-aware planning} (enforce discovery-before-implementation phases; estimated +5--10\% PASS).

\finding{Input-side prompt engineering (long vs.\ short) yields up to +25.0pp pass rate improvement for capable agents. The remaining 75.0\% non-pass gap requires output-side and process-side harnesses---particularly fix-cycle loops and enforced test generation---that no current configuration implements.}
