\section{Introduction}
\label{sec:introduction}

Software engineering teams are rapidly integrating AI into their development workflows; at Huawei, for example, multiple stack-level teams (CANN, MindSpeed, Kubernetes) are actively deciding whether to adopt coding agents for production feature work in the coming planning cycle, which motivates the evaluation we report here.
Industry surveys report that over 80\% of professional developers now use AI-assisted coding tools~\cite{github2024survey}, and frontier coding agents---autonomous systems that navigate codebases, write code, and run tests---have demonstrated remarkable progress on established benchmarks.
On SWE-bench Verified, the most widely cited evaluation for coding agents, top systems now resolve over 70\% of tasks~\cite{jimenez2024swebench}.
These results have generated significant enthusiasm for deploying coding agents on increasingly ambitious engineering tasks, including full feature implementation in enterprise codebases.

However, SWE-bench and similar benchmarks overwhelmingly evaluate \emph{bug fixing}: localized edits to 1--3 files, typically under 50 lines of change, in well-documented open-source Python repositories.
Feature implementation in industrial settings is a fundamentally different class of task.
It requires API design, feature gate lifecycle management, cross-module coordination, backward compatibility constraints, test suite construction, and adherence to project-specific conventions---engineering activities that demand architectural reasoning, not just code generation.
A Kubernetes feature gate, for example, touches API types, validation, defaulting, conversion, controllers, and dozens of generated files across 50--100 files; a CANN operator implementation requires host code, device kernels, tiling engines, and framework integration layers spanning C++, AscendC, and Python.
No systematic evaluation exists for how coding agents perform on this class of tasks in production codebases.

We present the first industrial-scale capability assessment of coding agents on feature implementation tasks.
Our benchmark comprises 62 tasks: 12 industrial tasks drawn from four Huawei production repositories---Kubernetes (Go), CANN operator kernels (C++/AscendC), MindSpeed training framework (Python), and torch\_npu (Python)---spanning three complexity tiers from single-file fixes to 98-file architectural changes, plus 50 multi-file tasks from the CapBench open-source suite (Kafka, Airflow, CPython, Django, Spring, and others).
We evaluate four commercially available agent configurations across 496 experiments---each independently scored by four LLM evaluators (Claude, Codex, Cursor, GLM) for cross-validation via majority-vote verdicts with conservative tie-breaking---and find a clear answer: \textbf{no.}
The best configurations, Claude Opus~4.6 and GPT-5.4 (Codex), achieve only 2.4\% PASS; Cursor Composer-2 and OpenCode GLM-5.1 each reach 0.8\%.
Overall, only 8 of 496 runs pass (1.6\%), and \emph{all four} agents fail at 0\% on Kubernetes-scale refactors.

This paper makes five contributions aimed at industrial decision makers as
well as benchmark designers:

\begin{enumerate}
  \item \textbf{An industrial feature-work evaluation setting.}
    We define the deployment question, task scope, and production-equivalence criterion for judging whether a coding agent can replace the first draft of a human feature PR (\S\ref{sec:industrial-context}).

  \item \textbf{A controlled multi-repository benchmark.}
    We evaluate 62 real feature-implementation tasks (12 Huawei enterprise + 50 CapBench open-source), spanning six languages and three complexity tiers, with merged production code as the reference implementation (\S\ref{sec:study-design}).

  \item \textbf{A conservative multi-evaluator protocol.}
    Each generated diff is scored by four evaluator models under a three-dimensional rubric, enabling us to report both majority PASS rates and judge-disagreement statistics rather than relying on a single LLM-as-judge (\S\ref{sec:eval-framework}, \S\ref{sec:results-stats}).

  \item \textbf{Empirical capability boundaries.}
    Across 496 runs, only 8 pass (1.6\%); all Kubernetes-family attempts fail; and richer prompts improve Claude/Codex but do not overcome the systems-code cliff (\S\ref{sec:results}).

  \item \textbf{Failure analysis and deployment guidance.}
    We show that non-PASS outcomes are dominated by incomplete scope coverage rather than pure semantic misunderstanding, and translate this into concrete harness and routing recommendations for near-term production use (\S\ref{sec:root-cause}, \S\ref{sec:industrial-deployment}).
\end{enumerate}
