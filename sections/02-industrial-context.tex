\subsection{Coding-Agent Benchmarks}

SWE-bench~\cite{jimenez2024swebench,chowdhury2024swebenchverified} established real-repository issue resolution as the dominant evaluation paradigm for coding agents. Follow-on work improves data quality, multilingual coverage, and live collection~\cite{aleithan2024swebenchplus,badertdinov2025swerebench,multiswebench,zan2024swebenchjava,swebenchlive}. These benchmarks are valuable for bug repair, but their acceptance condition is usually an executable oracle or issue-level resolution. Industrial feature PRs require a stricter notion of merge readiness: the implementation must be complete, idiomatic for the repository, and aligned with the feature's architectural surface.

Feature-oriented benchmarks move closer to our setting. SWE-bench Pro~\cite{swebenchpro2025}, SWE-EVO~\cite{sweevo}, FeatBench~\cite{featbench2025}, FeatureBench~\cite{zhou2026featurebench}, NL2Repo-Bench~\cite{ding2025nl2repo}, and Web-Bench~\cite{xu2025webbench} evaluate longer-horizon changes. Our main corpus differs in scale and criterion: the median task changes 57 GT files and 6{,}905 LOC, and SE-PASS requires production-equivalent behavior and structure rather than test passing alone.

Table~\ref{tab:benchmark-positioning} summarizes this positioning. The key distinction is not that prior benchmarks are small or easy; rather, they usually make the success condition more localized than the decision faced by an industrial reviewer. A patch can pass tests, repair an issue, or generate a plausible repository skeleton while still being unsuitable as a feature PR because it misses cross-module registrations, changes the wrong abstraction layer, or omits migration and compatibility behavior. Our benchmark is designed around that reviewer-facing boundary.

\subsection{Feature-Level and Long-Horizon Tasks}

We use \emph{long-horizon} in the sense crystallized by recent coding-agent benchmarks~\cite{swebenchpro2025,sweevo,featbench2025}: a task is long-horizon when the agent must translate a high-level requirement into a multi-step implementation plan, coordinate edits across modules, and sustain intent over a prolonged trajectory with sparse feedback. In our setting, the term also carries an industrial constraint: the output is judged as a candidate PR, not as an isolated code snippet. A valid solution therefore includes tests, repository-specific integration points, and compatibility with existing architecture.

This framing makes task scope a first-class variable. File count and line count are useful descriptive signals, but they are incomplete proxies for difficulty: a one-file compiler intrinsic can require specialized domain knowledge, while a many-file generated-code update may be mostly mechanical. We therefore separate \emph{scale} from \emph{coordination}. Scale captures the amount of code touched by the merged PR. Coordination captures whether the agent must discover and modify multiple subsystems whose behavior only composes at runtime, such as scheduler policies, API registration tables, wire formats, or generated bindings.

\subsection{Coding-Agent Systems and Industrial Adoption}

Research systems such as SWE-agent~\cite{harness2025}, Agentless~\cite{agentless}, AutoCodeRover~\cite{zhang2024autocoderover}, SWE-Fixer~\cite{xie2025swefixer}, and multi-agent coding frameworks~\cite{li2025swedebate,dong2023selfcollab,costa2026agentspawn} show that scaffolding, planning loops, and validation harnesses materially affect agent behavior. Industrial reports and adoption studies likewise suggest that narrow tasks land more reliably than broad changes~\cite{mathai2024passerine,autobug,logsage,alshahwan2024testgen,watanabe2025agenticgithub,ehsani2026failedprs,robbes2026adoption}. This motivates our product-level framing: each evaluated agent is a deployed combination of model, tools, retrieval, and execution harness.

The same observation also limits causal claims. A commercial coding agent is not only a model checkpoint; it is an integrated product with a terminal interface, search tools, patch application behavior, memory, retry strategies, and hidden validation heuristics. Our results therefore speak to the products that an engineering team can actually deploy. They do not isolate whether a failure is caused by the foundation model, the tool scaffold, or the interaction strategy.

\subsection{LLM-as-Judge and Human Calibration}

Open-ended code evaluation increasingly uses LLM judges, but prior work shows that single judges can be unstable and biased~\cite{gu2024llmjudgesurvey,li2024llmsjudgessurvey,crupi2025llmjudge,thakur2024judgingjudges,jung2024trustescalate}. Multi-judge panels improve robustness~\cite{chen2025multiagentjudge,li2025metajudges,loi2025autobench}, while domain calibration remains important for software-engineering tasks~\cite{wang2025canllmsreplace,zhou2025llmjudgese,wu2024codeeval}. Our evaluation therefore combines deterministic HW/GT coverage reports, four LLM evaluators, conservative tie-breaking, a leave-one-family-out bias check, and blinded human calibration on the 12-task industrial subset.
